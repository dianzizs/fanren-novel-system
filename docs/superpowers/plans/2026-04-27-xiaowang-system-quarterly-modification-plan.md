# 小王一号项目季度修改规划（按 ai_dev_rules 范式）

> 规划时间：2026-04-27  
> 规划周期：1 个季度（12 周）  
> 适用目录：`D:/pythonProject/小王一号/小王一号/`

## 0. 规划假设与边界

### 0.1 明确假设
- 本规划聚焦工程化改造与稳定交付，不改变项目核心业务目标（小说问答/续写/图谱）。
- 允许在现有架构上做渐进式调整，优先沿用已有模块边界，不做一次性重构。
- 现有技术栈保持 Python + FastAPI + 本地文件存储 + 向量检索，不引入重型基础设施（如分布式数据库）作为本季度前置条件。

### 0.2 不在本季度范围
- 大规模前端重写（如迁移到全新前端框架）。
- 跨云部署体系重构。
- 与核心改造无关的“顺手优化”。

---

## 1. 项目现状基线（来自当前代码事实）

### 1.1 关键结构与入口
- API 入口：`novel_system/api.py`
- 服务聚合：`novel_system/service.py`
- 索引流水线：`novel_system/index_pipeline.py`
- 仓储层：`novel_system/repository.py`
- 运行脚本：`scripts/run_api.py`、`scripts/build_index.py`、`scripts/run_eval.py`
- 测试目录：`tests/`

### 1.2 当前可见问题（直接影响改造优先级）
1. 文档与代码存在漂移（README 中部分结构描述与当前实现不完全一致）。
2. 评测结果默认路径与仪表盘读取路径历史上有不一致风险（需统一并固化）。
3. 依赖较重（`torch/openvino/optimum-intel/faiss-cpu` 等），跨机器安装与运行稳定性风险高。
4. `pyproject.toml` 目前主要承载 pytest 配置，工程元数据与任务编排能力偏弱。
5. 仓库存在较多并行变更痕迹，改造需要先建立“基线冻结 + 小步提交”机制。

### 1.3 改造总目标（季度）
- 建立“可持续迭代”的工程骨架：文档、测试、脚本、发布与回滚流程闭环。
- 提升稳定性与可观测性：关键路径可测、可追踪、可定位。
- 降低环境不确定性：依赖安装、启动与评测流程可重复。

---

## 2. 按 ai_dev_rules 的执行原则映射

1. **先理解再改造**：每个里程碑启动前先产出当前状态快照（模块、配置、脚本、接口）。
2. **最小变更**：每次改动必须能回答“为何必要”。不做无目标重构。
3. **分层一致**：API/Service/Repository 责任边界不跨层。
4. **渐进交付**：按“基线 -> 质量 -> 能力”三阶段推进，每阶段都可独立验收。
5. **可追溯**：关键决策和变更记录写入 docs，并与代码同版本演进。
6. **可回滚**：任何生产化脚本改造都配套回滚步骤。

---

## 3. 改造路径比较（2-3 方案）

### 方案 A：全量系统性改造（推荐）
- 内容：文档基线、测试策略、依赖治理、接口一致性、发布回滚一起推进。
- 优点：季度结束时形成完整工程闭环。
- 缺点：前 2-3 周投入偏大，对执行纪律要求高。

### 方案 B：稳定交付优先
- 内容：先测试与脚本，再做小幅架构/文档修复。
- 优点：短期收益快。
- 缺点：长期债务残留，下一阶段仍需补治理。

### 方案 C：功能迭代优先
- 内容：围绕新功能开发优化流程，工程治理穿插进行。
- 优点：业务可见度高。
- 缺点：基础治理容易被功能节奏挤压。

**结论：采用方案 A。** 当前项目已具备多模块能力，主要瓶颈在工程一致性与可持续交付，系统性改造收益最高。

## 4. 季度路线图（12 周）

## 阶段 I（第 1-4 周）：基线收敛与规范对齐

### 目标
建立统一“事实来源”，消除文档/脚本/配置漂移，固定最小可运行闭环。

### 任务包
1. **项目事实清单固化**
   - 输出模块清单、入口链路、运行脚本说明。
   - 文档位置：`docs/` 下新增或更新基线文档（与 README 保持一致）。
2. **README 与真实结构对齐**
   - 更新“项目结构”“运行步骤”“评测路径”部分为当前真实实现。
3. **评测路径统一**
   - 统一 `scripts/run_eval.py` 与服务层读取的评测报告路径定义。
4. **配置项梳理**
   - 明确 `.env.example` 中必须项、可选项、默认值来源。
5. **脚本可重复性检查**
   - 明确三个核心脚本的输入/输出、失败行为和日志落点。

### 产物
- 基线文档（模块与流程图谱）
- 对齐后的 README
- 统一后的评测路径约定
- 配置项说明清单

### 验收标准
- 新同学可在不查历史聊天记录的情况下完成安装、索引、启动、评测全流程。
- README 与代码结构不再出现关键路径级别矛盾。

---

## 阶段 II（第 5-8 周）：质量闭环与风险治理

### 目标
形成“改动可测、问题可回归、故障可定位”的最低工程保障。

### 任务包
1. **测试分层策略落地**
   - 单测：核心纯逻辑（planner/validator/search 规则）
   - 集成测试：API 关键端点与索引-检索主流程
   - 回归测试：P0 问题样例集固化
2. **关键路径用例补齐**
   - 重点覆盖：问答、续写、图谱、书籍导入、索引构建、评测仪表盘数据读取。
3. **错误分类与日志规范**
   - 区分用户输入错误、业务校验错误、外部依赖错误。
   - 日志字段统一（请求ID、book_id、阶段、耗时、错误类型）。
4. **安全基线治理**
   - 上传文件路径与文件名安全校验。
   - 敏感配置仅来自环境变量，不写死在代码与日志。
5. **提交与审查节奏**
   - 约束小步提交；每个 PR 对应可验证目标。

### 产物
- 测试策略文档 + 关键测试用例集
- 错误码/错误分类与日志字段约定
- 安全基线检查清单

### 验收标准
- 关键功能点均有至少一条自动化回归路径。
- 已知高频问题可在日志中快速定位到模块层面。

---

## 阶段 III（第 9-12 周）：工程化增强与发布韧性

### 目标
把“能开发”升级为“能稳定发布与迭代”。

### 任务包
1. **依赖治理与运行矩阵**
   - 形成基础运行集与增强运行集（按环境能力分层安装）。
   - 明确 CPU-only 与加速环境差异。
2. **发布流程标准化**
   - 形成发布检查清单（配置、测试、数据目录、回滚点）。
3. **回滚与恢复演练**
   - 对索引产物、评测报告、配置变更建立回滚预案。
4. **性能与稳定性基线**
   - 建立问答/索引关键耗时基线，记录波动阈值。
5. **文档闭环**
   - 将阶段成果回写到 README 与 docs，形成“单一真相源”。

### 产物
- 依赖分层方案与安装说明
- 发布/回滚 runbook
- 性能基线记录
- 最终季度改造总结

### 验收标准
- 新版本发布可按清单执行并在异常时可回退。
- 核心流程耗时与失败率具备可观测基线。

## 7. 风险清单与应对

1. **并行变更冲突**
   - 应对：阶段内冻结高风险目录；每周合并窗口。
2. **依赖安装不稳定**
   - 应对：分层依赖 + 环境矩阵 + 锁定关键版本。
3. **测试维护成本上升**
   - 应对：优先关键路径，不追求“全覆盖率”。
4. **文档再次漂移**
   - 应对：将文档更新纳入 PR 完成定义（DoD）。
